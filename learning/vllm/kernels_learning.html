0. 前言：为何 GPU Kernel 是 LLM 推理的核心瓶颈

大语言模型推理的计算图可以粗略地分成两类操作：Attention（注意力计算）和 GEMM（矩阵乘法）。这两类操作的性能特征截然不同，需要完全不同的优化策略。理解这一区别是理解 vLLM 所有 GPU Kernel 设计决策的起点。

Roofline 模型

GPU 性能受两个天花板限制：计算峰值（Peak FLOPS）和内存带宽（Memory Bandwidth，BW）。Roofline 模型用算术强度（Arithmetic Intensity，AI）来刻画一个算法究竟是哪个瓶颈：

$$\text{Performance} = \min\!\left(\text{Peak FLOPS},\; AI \times BW\right)$$$$AI = \frac{\text{FLOPs}}{\text{Bytes读写}}$$

Prefill 阶段（处理输入 prompt）对长度为 $S$ 的序列做完整 Attention，复杂度 $O(S^2 d)$，算术强度高，属于 compute-bound。Decode 阶段（逐 token 生成）每步仅计算一个 token 对全部历史 KV 的 Attention，复杂度 $O(Sd)$，算术强度极低，属于 memory-bandwidth-bound。这就是为什么 Decode 阶段的 Attention Kernel 优化重点是减少 HBM 读写，而非提高 FLOPS 利用率。

2. FlashAttention Tier 1

核心文件：vllm/v1/attention/backends/flash_attn.py（47KB）、csrc/attention/paged_attention_v1.cu、csrc/attention/paged_attention_v2.cu

2.1 标准 Attention 的内存问题

标准缩放点积注意力（SDPA）公式为：

$$O = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d}}\right)V$$

对于序列长度 $N$，这需要在 HBM 中实例化 $N \times N$ 的注意力矩阵 $S = QK^\top / \sqrt{d}$，峰值显存 $O(N^2)$。当 $N = 128K$ 时（长上下文场景），仅这一张矩阵就需要约 64GB FP16 存储——超过 A100 80GB 的全部显存。FlashAttention 的核心贡献正是通过分块计算彻底消除这张矩阵。

2.2 Online Softmax：分块计算的数学基础

Online Softmax 维护三个在线统计量 $m_i$（当前最大值）、$\ell_i$（归一化分母）和 $O_i$（累积输出），在逐块处理 K/V 时通过递推更新：

$$m_i^{(\text{new})} = \max\!\left(m_i^{(\text{old})},\; \max_j s_{ij}\right)$$ $$\ell_i^{(\text{new})} = e^{\,m_i^{(\text{old})} - m_i^{(\text{new})}} \ell_i^{(\text{old})} + \sum_j e^{\,s_{ij} - m_i^{(\text{new})}}$$ $$O_i^{(\text{new})} = \underbrace{\frac{\ell_i^{(\text{old})}}{\ell_i^{(\text{new})}} e^{\,m_i^{(\text{old})}-m_i^{(\text{new})}}}_{\text{修正旧 }O} \cdot O_i^{(\text{old})} + \frac{1}{\ell_i^{(\text{new})}}\sum_j e^{\,s_{ij}-m_i^{(\text{new})}} v_j$$

可以数学证明：遍历所有 K/V 块后，$O_i^{(\text{final})} = \text{softmax}(QK^\top/\sqrt{d})V$，结果与全量计算等价，但峰值 HBM 用量仅 $O(Nd)$（去掉了 $N^2$ 项）。

2.3 SRAM Tiling 策略

FlashAttention 将 $Q$ 按行切成块（行块大小 $b_r$），将 $K/V$ 按列切成块（列块大小 $b_c$），每次将一对 $(K_{b_c}, V_{b_c})$ 从 HBM 加载到 SRAM 做内循环。SRAM 总占用：

$$\text{SRAM usage} = O(b_r \cdot d + b_c \cdot d)$$

$b_r$ 和 $b_c$ 由片上 SRAM 大小决定，典型值在 SM80（A100）为 $b_r = b_c = 64$ 或 128。

动画 A1：FlashAttention SRAM Tiling 逐步演示

2.4 Paged KV Cache

vLLM 采用 PagedAttention 机制，将 KV Cache 按固定大小的物理块（block）管理，逻辑序列通过 block table 映射到非连续的物理内存：

$$\text{block\_table}[\text{seq\_id}][\text{logical\_block}] = \text{physical\_block\_id}$$

FlashAttention 后端支持 MultipleOf(16) 的任意 block size；混合模型（Mamba+Attention）因 FA NaN 传播问题限制为 [16, 32, 64]。

⚠ 已知 Bug：当混合模型使用 FP32 SSM cache 时，FlashAttention 某些 block size 会触发 NaN 传播。参见 flash-attention#1974。源码注释：flash_attn.py:91

2.5 FP8 KV Cache 量化

当 kv_cache_dtype="fp8" 时，K/V 以 E4M3 格式存储，Attention 计算前按 scale 还原：

$$K_{\text{restored}} = K_{\text{fp8}} \cdot s_K, \quad V_{\text{restored}} = V_{\text{fp8}} \cdot s_V$$

FlashInfer 后端的 _trtllm_prefill_attn_kvfp8_dequant Triton kernel 实现了融合的 FP8 反量化+Attention 计算。

2.6 Cascade Attention（共享前缀优化）

当多个序列共享相同前缀（如 system prompt）时，FlashAttention 后端将 Attention 分为两阶段：

  1. Prefix 阶段：所有序列共用已缓存的 prefix KV，计算 $O_{\text{prefix}}, \text{LSE}_{\text{prefix}}$
  2. Suffix 阶段：各序列独立计算 $O_{\text{suffix}}, \text{LSE}_{\text{suffix}}$

最后通过 merge_attn_states 将两阶段结果按 LSE 加权合并：

$$O = \frac{\text{LSE}_p \cdot O_p + \text{LSE}_s \cdot O_s}{\text{LSE}_p + \text{LSE}_s} \quad \text{（对数域合并）}$$
# flash_attn.py - FlashAttentionBackend

4.3 SM90+ Persistent Kernel

在 Hopper（SM90）及以上硬件，TRTLLM-GEN 使用 Persistent Kernel 设计：threadblock 在整个推理步骤期间持续运行，从队列中拉取工作，而非为每个 sequence 单独 launch kernel，大幅降低 kernel launch 延迟。

5.1 MLA 数学动机（DeepSeek-V2 创新）

标准 Multi-Head Attention（MHA）的 KV Cache 大小随序列长度线性增长：

$$\text{MHA KV Cache size} = 2 \cdot n_{\text{heads}} \cdot d_{\text{head}} \cdot L \cdot \text{bytes}$$

对于 DeepSeek-V3（128 heads, d_head=128, fp16），每 token 需 2×128×128×2 = 65536 字节 ≈ 64KB，100K token 上下文需 6.4GB——仅 KV Cache 就已接近 GPU 显存上限。

MLA（Multi-head Latent Attention）通过低秩分解将 KV 压缩为潜变量 $c^{KV}$：

$$c^{KV} = W_{DKV} x \in \mathbb{R}^{d_c}, \quad d_c \ll n_{\text{heads}} \cdot d_{\text{head}}$$ $$K = W_{UK} c^{KV} \in \mathbb{R}^{n_{\text{heads}} \cdot d_{\text{head}}}, \quad V = W_{UV} c^{KV}$$

KV Cache 只存低维潜变量 $c^{KV}$，解压在 Attention 计算时即时进行（"Absorption Trick"）。

位置编码（RoPE）部分需要特殊处理——不能吸收进权重矩阵，需单独保留：

$$Q = \left[W_{UQ}c^Q \;;\; q_{pe}\right], \quad K = \left[W_{UK}c^{KV} \;;\; k_{pe}\right]$$

实际 KV Cache 存储格式：$[c^{KV},\; k_{pe}]$ 拼接，对应代码中的 kv_c_and_k_pe_cache。

6.1 两阶段 Flash Decoding

Triton Attention 采用 Split-K（Flash Decoding）策略，将 KV 序列切成多段并行处理：

# triton_attn.py
MIN_LAUNCH_GRID_SIZE_2D = 128
NUM_PAR_SOFTMAX_SEGMENTS = 16   # 并行 softmax 分段数

Stage 1（_fwd_kernel_stage1）：每段独立计算局部 Attention，输出局部 $O_s$ 和 $(m_s, \ell_s)$

Stage 2：跨段归约：

$$m^* = \max_s m_s, \quad \ell^* = \sum_s e^{m_s - m^*} \ell_s, \quad O = \sum_s \frac{e^{m_s - m^*} \ell_s}{\ell^*} O_s$$

7. GEMM 基础与硬件演进

7.1 矩阵乘法数学模型

标准 GEMM（General Matrix Multiply）：

$$C[m,n] = \sum_{k=0}^{K-1} A[m,k] \cdot B[k,n]$$

量化 GEMM（W8A8）：权重与激活分别量化后做整数矩阵乘，最后在 epilogue 反量化：

$$C = \text{dequant}(A_{\text{int8}} \cdot B_{\text{int8}}) \cdot s_A \cdot s_B + \text{bias}$$

8.1 量化 GEMM 数学

$$C = \text{dequant}(A_{q} \cdot B_{q}) \cdot s_A \cdot s_B + \text{bias}$$

支持两种 scale 粒度：

  • Per-tensor：$A$ 和 $B$ 各一个全局 scale，计算最快
  • Blockwise（MXFP8）：每 block 共享一个 scale，精度更高：$\hat{x}_i = \text{fp8\_decode}(q_i) \cdot s_{b(i)}$

8.2 SM 版本分发链

// scaled_mm_entry.cu - SM 版本自动选择
void cutlass_scaled_mm(torch::Tensor& out, const torch::Tensor& a,
                       const torch::Tensor& b,
                       const torch::Tensor& scale_a,
                       const torch::Tensor& scale_b,
                       const std::optional<torch::Tensor>& bias) {
  int sm = at::cuda::getCurrentDeviceProperties()->major * 10
         + at::cuda::getCurrentDeviceProperties()->minor;
  if      (sm >= 120) cutlass_scaled_mm_sm120(out, a, b, scale_a, scale_b, bias);
  else if (sm >= 100) cutlass_scaled_mm_sm100(out, a, b, scale_a, scale_b, bias);
  else if (sm >= 90)  cutlass_scaled_mm_sm90 (out, a, b, scale_a, scale_b, bias);
  else if (sm >= 89)  cutlass_scaled_mm_sm89 (out, a, b, scale_a, scale_b, bias);
  else if (sm >= 80)  cutlass_scaled_mm_sm80 (out, a, b, scale_a, scale_b, bias);
  else                cutlass_scaled_mm_sm75 (out, a, b, scale_a, scale_b, bias);
}

9.1 执行流程

  1. 解包：INT4 packed 权重（每字节存两个 INT4 值）解包为临时 INT8 格式
  2. Tensor Core：以 FP8 精度执行 mma 指令（INT4→FP8 转换在片上完成）
  3. Epilogue：应用 per-column（per-output-channel）scale 反量化，输出 BF16/FP16
$$C_{f16} = \text{epilogue}(A_{fp8} \cdot \text{unpack}(B_{int4})) \cdot s_A \cdot s_{B,\text{per-col}}$$

9.2 MoE 分组 MM

w4a8_grouped_mm_entry.cu 支持每个专家独立的问题规模（Grouped GEMM），每个专家的 K/N/scale 可以不同，通过 CUTLASS 的 Grouped GEMM 实现高效并行。

10. FP4/NVFP4 GEMM Tier 2

核心文件：csrc/libtorch_stable/quantization/fp4/nvfp4_scaled_mm_kernels.cu（14KB）、nvfp4_scaled_mm_sm120_kernels.cu（13KB）

10.1 NVFP4 格式规范

NVIDIA FP4（E2M1 格式）：1 位符号 + 2 位指数 + 1 位尾数，共 4 位。可表示的非零值集合：

$$\{±0.5, ±1, ±1.5, ±2, ±3, ±4, ±6\} \cup \{0\}$$

单独使用 FP4 表示范围太窄，因此每 16 个 FP4 元素共享一个 FP8 E4M3 缩放因子（blockwise）：

$$\hat{x}_i = \text{fp4\_decode}(q_i) \cdot \text{fp8\_decode}(s_{b(i)}), \quad b(i) = \lfloor i/16 \rfloor$$

这等价于 OCP MX 标准中的 MXFP4，但 NVIDIA 实现有硬件特定优化。

10.2 Blackwell SM120 FP4 Tensor Core

Blackwell B200（SM120）引入专用 FP4 Tensor Core 指令，理论峰值吞吐约为 FP8 的 2 倍（~14 PFLOPS vs ~7 PFLOPS FP8）。nvfp4_scaled_mm_sm120_kernels.cu 专门针对 SM120 优化，使用新的 tcgen05.mma 指令族。

11. CuTeDSL 与 CUTLASS 3.x 张量抽象 Tier 2

CuTe（CUTLASS Tensor Extension）是 CUTLASS 3.x 引入的张量描述语言，用于精确且可组合地表达 CUDA Kernel 中的 tile 形状、步长和数据访问模式。

11.1 Shape 与 Stride 抽象

CuTe 使用嵌套元组（hierarchical tuples）描述多维张量，编译期常量用 _N 表示：

// 来自 sm100_cutlass_mla_kernel.cu
// Tile: (H=128 heads, N=128 KV-seq, D=(512 latent, 64 rope))
using TileShape = Shape<_128, _128, Shape<_512, _64>>;

// Stride: (H_stride, D_stride=1_contiguous, B_stride)
using StrideQ = cute::tuple<int64_t, _1, int64_t>;  // H D B

// 访问 Q[h, d, b]：
// offset = h * stride_H + d * 1 + b * stride_B

11.2 TMA（Tensor Memory Accelerator）

Hopper（SM90）引入的 TMA 硬件单元可以异步地将多维张量从 HBM 批量拷贝到 SRAM，不消耗线程计算资源。CUTLASS 3.x 通过 TmaWarpspecialized kernel 模板封装 TMA，实现计算与内存访问的完全流水线化：

// CUTLASS 3.x TMA warpspecialized kernel 结构示意
// Warp 0: 专门做 TMA 加载（consumer）
// Warp 1-3: 专门做 TC 计算（producer）
// 通过 __pipeline_wait_prior / arrive_and_wait 同步
using FmhaKernel = Sm100FmhaMlaKernelTmaWarpspecialized<...>;

11.3 CuTeDSL：Python 端 DSL

CuTeDSL 是 CuTe 的 Python 绑定，允许在 Python 中用 CuTe 语法描述 Kernel，通过 MLIR 编译到高效 CUDA 代码。vLLM 的部分自定义 Kernel 开始采用此方式开发。

12. Marlin W4A16 GEMM Tier 1

核心文件：csrc/quantization/marlin/marlin.cu、marlin_mma.h、marlin_template.h

支持：AWQ、GPTQ 量化权重，W4A16/W8A16，split-K

12.1 数学模型

权重以 INT4 packed 格式存储，激活为 FP16/BF16，矩阵乘时即时反量化：

$$y = \hat{W} x, \quad \hat{W}_{ij} = s_{g(i,j)} \cdot \left(q_{ij} - z_{g(i,j)}\right)$$

其中 $g(i,j)$ 为 group 索引，$s$ 为 scale，$z$ 为 zero-point。group size 通常为 128。

12.4 Split-K 并行归约

对 M 很小（Decode 阶段常见）的情况，沿 K 维并行以提高 GPU 利用率：

$$y[m,n] = \sum_{s=0}^{S-1} \sum_{k \in \text{slice}_s} \hat{W}[k,n] \cdot x[m,k]$$

输出形状 [split_k_iters, M, N]，后归约为 [M, N]。

12.5 列置换与权重重打包

列置换（permute_cols_kernel）：离线预处理，将权重列重排以对齐 ldmatrix 的访问模式，消除 shared memory bank conflict。

权重重打包：

  • awq_marlin_repack.cu：将 AWQ 量化格式（按列分组）转换为 Marlin interleaved 布局
  • gptq_marlin_repack.cu：将 GPTQ 格式（行主序 INT4 packed）转换为 Marlin 布局

Marlin interleaved 布局的核心思想：使每个 warp 的 32 个线程在执行 ldmatrix 时访问的权重元素恰好分布在不同的 bank，实现零 bank conflict 的权重加载。

13. MoE 路由 Kernels Tier 1

核心文件：csrc/moe/topk_softmax_kernels.cu（36KB）、csrc/moe/grouped_topk_kernels.cu（43KB）、csrc/moe/moe_permute_unpermute_op.cu（8.8KB）、csrc/moe/moe_align_sum_kernels.cu（30KB）

13.1 MoE 路由数学

给定 token 特征 $x$，router 产生 logits 并选择 Top-K 专家：

$$r = x W_{\text{router}} \in \mathbb{R}^{n_{\text{experts}}}$$ $$\text{expert\_ids} = \text{TopK}\!\left(\text{softmax}(r),\; k\right)$$ $$g_i = \frac{e^{r_i}}{\sum_{j \in \text{TopK}} e^{r_j}} \quad \text{（归一化门控权重）}$$

部分模型（如 DeepSeek-V3）使用 sigmoid 替代 softmax：

$$g_i = \frac{\sigma(r_i)}{\sum_{j \in \text{TopK}} \sigma(r_j)}$$

13.2 TopK Softmax Kernel

// csrc/moe/topk_softmax_kernels.cu
enum class ScoringFunc { SCORING_SOFTMAX = 0, SCORING_SIGMOID = 1 };

template <int TPB, typename InputType>
__launch_bounds__(TPB) __global__
void moeSoftmax(const InputType* input, const bool* finished,
                float* output, const int num_cols) {
  // 每个 block 处理一个 token 的所有 expert logits
  using BlockReduce = cub::BlockReduce<float, TPB>;
  __shared__ typename BlockReduce::TempStorage temp_storage;

  float local_max = -FLT_MAX;
  // Step 1: 找最大 logit（数值稳定性）
  for (int i = threadIdx.x; i < num_cols; i += TPB)
    if (!finished || !finished[blockIdx.x])
      local_max = fmaxf(local_max, (float)input[blockIdx.x * num_cols + i]);
  float global_max = BlockReduce(temp_storage).Reduce(local_max, cub::Max());

  __syncthreads();
  // Step 2: 计算 exp(logit - max)，求和后归一化
  float local_sum = 0.f;
  for (int i = threadIdx.x; i < num_cols; i += TPB)
    local_sum += expf((float)input[blockIdx.x * num_cols + i] - global_max);
  float global_sum = BlockReduce(temp_storage).Sum(local_sum);
  // Step 3: 写入归一化 softmax 输出
}

FlashAttention NaN 传播（混合模型）
来源：flash_attn.py:91 注释 | 参考：flash-attention#1974

当 Mamba+Attention 混合模型使用 FP32 SSM cache（mamba_ssm_cache_dtype="float32"）时，FlashAttention 在某些 block size（如 128）下会产生 NaN。vLLM 临时限制这类模型的 block size 为 [16, 32, 64]。根本原因尚在 FlashAttention 上游调查中。

CUTLASS MLA SM100 Kernel 要求 CUDA ≥ 12.4
来源：csrc/attention/mla/sm100_cutlass_mla_kernel.cu:37-48

Blackwell SM100 CUTLASS MLA Kernel 依赖 CUDA 12.4+ 的 TMA 和 warpspecialized 功能。CUDA < 12.4 时 kernel 以 TORCH_CHECK(false, ...) 主动报错，不会静默失败。用户需确保 Blackwell GPU 配套 CUDA ≥ 12.4 的环境。

FA2 vs FA3 CUDA Graph 兼容性差异
来源：flash_attn.py | 状态：进行中

FlashAttention v3（FA3，针对 Hopper 优化）与 vLLM CUDAGraph 捕获的兼容性不完整。FA2 在 SM80+ 上 CUDA Graph 完全支持；FA3 在某些配置下 graph capture 失败或运行时输出异常。vLLM 在 SM90 上默认根据版本检测在两者间切换。

Marlin Split-K 边界精度
来源：Marlin kernel | 状态：已知限制

W4A16 Marlin kernel 在特定 M/N/K 维度组合（尤其 K 不能被 block_k × split_k 整除时）可能出现边界 token 精度退化。解决方案是通过 padding 使 K 对齐到 tile 大小的整数倍，vLLM 在权重加载时自动处理 padding。

FP4 MoE Expert 量化：Blackwell 专属
来源：csrc/.../fp4/nvfp4_experts_quant.cu | 状态：SM120 限制

NVFP4 MoE Expert 量化目前仅支持 SM120（Blackwell B200/B100），在 SM90 及以下硬件上无法使用 FP4 量化 MoE 专家权重。尝试在非 SM120 硬件上使用此功能会回退到 FP8 或 BF16 路径。

FlashInfer workspace buffer 大小不足（已修复）
状态：已在 v0.13.x 修复

早期版本 FlashInfer workspace buffer 默认大小不足，在极长序列或大 batch 下会触发 OOM 或 kernel 错误。后更新为 2GB 固定常量 FLASHINFER_WORKSPACE_BUFFER_SIZE_BATCH_INVARIANT，同时支持通过环境变量 VLLM_FLASHINFER_WORKSPACE_BUFFER_SIZE 覆盖。
