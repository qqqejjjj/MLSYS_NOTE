# Radix Cache 技术分析笔记

Radix Cache 是 SGLang 及其精简版 Mini-SGLang 中的核心优化技术之一，旨在通过在请求之间重用 **KV Cache** 的共享前缀，显著减少 Prefill 阶段的计算开销并降低推理延迟。

## 1. 核心概念与硬件位置

在 LLM 推理中，许多请求（尤其是多轮对话、few-shot 任务）具有相同的前缀。传统的缓存管理（如简单的 LRU）通常以整个序列为单位，难以高效处理这种前缀重用。Radix Cache 将 KV Cache 管理抽象为一个 **基数树 (Radix Tree)** 结构。

### 1.1 Cache 在推理硬件中的位置

在讨论 Radix Cache 之前，我们需要明确 **KV Cache 存放在哪里**。

在 LLM 的推理过程中（尤其是自回归生成阶段），为了避免重复计算历史 token 的 Attention 矩阵，我们会将历史 token 的 Key (K) 和 Value (V) 张量保存下来，这就是 KV Cache。

- **物理存储位置**：KV Cache 绝大多数情况下存放在 **GPU 显存 (VRAM)** 中（高带宽显存 HBM）。因为 GPU 需要在每次生成新 token 时，极速读取这些历史信息来计算当前的 Attention。
- **Radix Cache 的角色**：Radix Cache 本身**并不是显存中的那块物理数据**，而是运行在 **CPU (Host)** 上的一个**逻辑管理器**。
  - **显存池**：GPU 显存中被划分出一个巨大的 Paged Attention 显存池（由无数个固定大小的 `page` 组成）。
  - **Radix Tree**：驻留在 CPU 内存中，负责记录 "哪段文本前缀 (Token IDs) 对应显存里的哪些物理 Page (indices)"。

**系统架构层次图**：
```
┌────────────────────────────────────────────────────────┐
│ CPU (Host)                                             │
│ ┌────────────────────────────────────────────────────┐ │
│ │ Radix Prefix Cache (逻辑层)                        │ │
│ │ 维护一棵 Radix Tree，进行快速的前缀字符串(token)匹配   │ │
│ │ 记录: 前缀 A -> 物理页 [0,1,2]                      │ │
│ └─────────────────────────┬──────────────────────────┘ │
│                           │ (控制指令 & 索引映射)      │
│ ┌─────────────────────────▼──────────────────────────┐ │
│ │ Paged Memory Allocator (分配器)                    │ │
│ │ 管理哪些显存页是空闲的，哪些是被占用的               │ │
│ └─────────────────────────┬──────────────────────────┘ │
└───────────────────────────┼────────────────────────────┘
                            │ (PCIe / NVLink)
┌───────────────────────────▼────────────────────────────┐
│ GPU (Device)                                           │
│ ┌────────────────────────────────────────────────────┐ │
│ │ 物理 KV Cache 显存池 (Physical VRAM)               │ │
│ │ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐            │ │
│ │ │Page0│ │Page1│ │Page2│ │Page3│ │Page4│ ...        │ │
│ │ └─────┘ └─────┘ └─────┘ └─────┘ └─────┘            │ │
│ │ *实际存放的是 Float16/Bfloat16 格式的 K 和 V 张量   │ │
│ └────────────────────────────────────────────────────┘ │
│ ┌────────────────────────────────────────────────────┐ │
│ │ Attention Kernel (如 FlashAttention/vLLM PagedAttn)│ │
│ │ 根据 CPU 传来的物理页索引，直接读取对应的张量进行计算│ │
│ └────────────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────────┘
```

### 1.2 核心定义

- **节点 (Node)**：代表一段连续的 token 序列及其对应的 KV Cache 在 GPU 物理显存中的页面索引（indices）。
- **边 (Edge)**：连接不同前缀的逻辑关系。

### 1.1 Radix Tree 结构示意图

```
                          ┌─────────────┐
                          │   ROOT      │
                          │  (ref=1)    │
                          └──────┬──────┘
                                 │
                    ┌────────────┴────────────┐
                    │                         │
            ┌───────▼──────┐          ┌──────▼───────┐
            │  Node A      │          │  Node B      │
            │ key: [1,2,3,4]│          │ key: [5,6,7,8]│
            │ val: [0,1,0,1]│          │ val: [2,3,2,3]│
            │ ref_count: 1  │          │ ref_count: 0  │
            └───────┬───────┘          └──────┬────────┘
                    │                         │
           ┌────────┴────────┐              (leaf)
           │                 │
    ┌──────▼─────┐    ┌─────▼──────┐
    │  Node A1   │    │  Node A2   │
    │key:[9,10,  │    │key:[11,12, │
    │     11,12] │    │     13,14] │
    │val:[4,5,4,5]│   │val:[6,7,6,7]│
    │ref_count:1 │    │ref_count:0 │
    └────────────┘    └────────────┘
```

**说明**：
- 每个节点存储一段 token 序列（key）和对应的 KV Cache 物理页面索引（value）
- ref_count > 0 表示该节点正被某个活跃请求使用，受保护不会被驱逐
- 子节点共享父节点的前缀

## 2. 核心代码组件分析 (基于 `python/minisgl/kvcache/radix_cache.py`)

### 2.1 `RadixTreeNode`
每个节点存储：
- `_key`: Token IDs 序列。
- `_value`: 对应的 KV Cache 页面索引。
- `ref_count`: 引用计数，用于追踪该前缀是否正被某些正在运行的请求使用。
- `timestamp`: 最后访问时间，用于 LRU 驱逐策略。

### 2.2 `RadixPrefixCache`
管理整棵树的核心类，实现以下关键逻辑：

#### (1) 前缀匹配 (`_tree_walk`)
当新请求到达时，系统会从根节点开始遍历：
- 使用 `key_fn`（通常基于 `page_size` 的第一个 token 或 tuple）快速查找子节点。
- 使用 `fast_compare_key` (C++ 实现) 高效对比 token 序列。
- **对齐限制**：匹配长度必须是 `page_size` 的倍数，这是为了配合 Paged Attention 的页面管理。

**key_fn 的作用**：
```python
# 当 page_size = 1 时
key_fn = lambda x: x[0].item()  # 使用第一个 token 的值作为字典键

# 当 page_size > 1 时 (例如 page_size=4)
key_fn = lambda x: tuple(x[:4].tolist())  # 使用前 4 个 token 组成的元组

# 这样父节点的 children 字典可以快速定位：
node.children[key_fn([1,2,3,4,...])] → 找到对应子节点
```

**遍历示例**：
```
输入: [1,2,3,4,9,10,11,12]
遍历过程:
1. ROOT.children[key_fn([1,2,3,4,...])] → 查找键 (1,2,3,4)
2. 找到候选节点，使用 fast_compare_key 逐位对比
3. 发现前 4 个完全匹配，继续查找
4. node.children[key_fn([9,10,11,12])] → 查找键 (9,10,11,12)
```

#### (2) 节点分裂 (`split_at`)
如果请求的输入与树中某个节点的键只有部分匹配，Radix Cache 会执行分裂操作：
- 将现有节点拆分为两部分：匹配的前缀部分和剩余部分。
- 这样可以最大化重用已有的 KV Cache。

**分裂过程可视化**：
```
分裂前：
    Parent
      │
      ▼
   ┌──────────────────┐
   │   Node A         │
   │ key: [1,2,3,4,   │
   │       5,6,7,8]   │
   │ val: [0,1]       │
   └──────────────────┘

分裂操作: split_at(4)  ← 前 4 个匹配

分裂后：
    Parent
      │
      ▼
   ┌──────────────────┐
   │   Node A' (新)   │  ← 提取公共前缀
   │ key: [1,2,3,4]   │
   │ val: [0]         │
   └────────┬─────────┘
            │
            ▼
   ┌──────────────────┐
   │   Node A'' (旧)  │  ← 原节点缩减
   │ key: [5,6,7,8]   │
   │ val: [1]         │
   └──────────────────┘
```

#### (3) 缓存插入 (`insert_prefix`)
当请求完成或生成新 token 后，系统会将新的前缀和对应的物理页面索引插入树中。
- 只有完成整页（`page_size`）的内容才会被缓存。

#### (4) 驱逐策略 (`evict`)
当显存不足时，系统需要释放 KV Cache：
- **引用计数**：`ref_count > 0` 的节点受保护，不能被驱逐。
- **LRU 策略**：系统收集所有 `ref_count == 0` 的叶子节点，根据 `timestamp` 使用堆（Heap）优先驱逐最久未使用的节点。

**驱逐过程可视化**：
```
驱逐前状态：
         ROOT (ref=1) ← 受保护
          │
    ┌─────▼──────┐
    │  Node A    │
    │ ref: 0     │ ← 可驱逐，但有子节点
    └──────┬─────┘
           │
      ┌────┴────┬──────────┐
      │         │          │
   ┌──▼───┐ ┌──▼───┐  ┌──▼───┐
   │Node B│ │Node C│  │Node D│
   │ref: 1│ │ref: 0│  │ref: 0│
   │ts:100│ │ts:50 │  │ts:80 │ ← timestamp
   └──────┘ └──────┘  └──────┘
            ↑ ↑        ↑
            │ │        │
        可驱逐叶子节点

驱逐步骤：
1. 收集所有 ref_count==0 的叶子: [Node C, Node D]
2. 按 timestamp LRU 排序（堆）: [Node C(50), Node D(80)]
3. 优先驱逐 Node C (最老)
4. 如需更多，继续驱逐 Node D
5. 若 Node A 变成叶子且 ref==0，也会进入驱逐候选
```

## 3. 性能优势

- **减少计算 (Compute Saving)**：命中前缀后，不需要对该部分进行 Prefill 计算（Attention 计算），直接从 KVCache 中读取。
- **低延迟 (Reduced Latency)**：TTFT (Time To First Token) 大幅缩短。
- **显存高效 (Memory Efficiency)**：通过树状结构避免了相同前缀在显存中的多次冗余存储。

```python
# 返回: prefix_len=0, node=ROOT

# 创建 Node A，挂在 ROOT 下

# ---- 请求 R2: [1,2,3,4,9,10,11,12] ----

# 触发 split_at(4)，返回 prefix_len=4

# 仅 Prefill 剩余 [9,10,11,12]，分配物理页 [2]
new_indices = allocate_pages(1)  # [2]

# 插入新分支
cache.insert_prefix([1,2,3,4,9,10,11,12], [0,2])
# 创建 Node B，挂在 Node A' 下
```

## 5. 实际案例：逐步演示 Radix Cache 工作流程

假设 **page_size = 4**（即每页存储 4 个 token 的 KV Cache），我们逐步演示以下场景。

### 场景设定
- **物理 KV Cache 池**：有 pages [0, 1, 2, 3, 4, 5, ...] 等待分配
- **初始状态**：树中只有 ROOT 节点

---

### **步骤 1：首个请求到达**
**请求 R1**：输入 token 序列 `[1, 2, 3, 4, 5, 6, 7, 8]`

#### 操作流程：
1. **match_prefix([1,2,3,4,5,6,7,8])**
   - 从 ROOT 开始遍历，未找到任何匹配
   - 返回：`prefix_len = 0`

2. **Prefill 阶段**
   - 系统为这 8 个 token 分配物理页：[0, 1]（每页存 4 个 token）
   - 计算 Attention 并存储 KV Cache

3. **insert_prefix([1,2,3,4,5,6,7,8], [0,1])**
   - 从 ROOT 开始，创建新节点
   - 插入长度对齐到 page_size：`align_down(8, 4) = 8`

#### 插入后树结构：
```
         ROOT
          │
    ┌─────▼─────┐
    │  Node A   │
    │ key: [1,2,3,4,5,6,7,8]
    │ val: [0,1] │  ← 物理页面 0 和 1
    │ ref: 1     │  ← 请求 R1 持有
    └───────────┘
```

---

### **步骤 2：第二个请求到达（共享前缀）**
**请求 R2**：输入 token 序列 `[1, 2, 3, 4, 9, 10, 11, 12]`（前 4 个 token 与 R1 相同）

#### 操作流程：
1. **match_prefix([1,2,3,4,9,10,11,12])**
   - 从 ROOT 出发，找到 Node A
   - 使用 `fast_compare_key` 对比：前 4 个 token 匹配
   - **发现部分匹配** → 触发 `split_at(4)`

2. **节点分裂 (Split)**
   - 将 Node A 分裂为两部分：
     - 新节点：`key=[1,2,3,4], val=[0]` (前缀)
     - 原节点：`key=[5,6,7,8], val=[1]` (剩余)

#### 分裂后树结构：
```
         ROOT
          │
    ┌─────▼──────┐
    │  Node A'   │  ← 新创建的前缀节点
    │ key: [1,2,3,4]
    │ val: [0]    │
    │ ref: 2      │  ← R1 和 R2 共同持有
    └──────┬──────┘
           │
    ┌──────▼──────┐
    │  Node A''   │  ← 原 Node A 的剩余部分
    │ key: [5,6,7,8]
    │ val: [1]    │
    │ ref: 1      │  ← 仅 R1 持有
    └─────────────┘
```

3. **继续处理 R2**
   - 前 4 个 token 已命中，无需 Prefill
   - 仅对剩余 `[9,10,11,12]` 进行 Prefill，分配物理页 [2]
   - 插入新节点

#### 最终树结构：
```
         ROOT
          │
    ┌─────▼──────┐
    │  Node A'   │
    │ key: [1,2,3,4]
    │ val: [0]    │
    │ ref: 2      │
    └──────┬──────┘
           │
      ┌────┴────┐
      │         │
┌─────▼──────┐ ┌▼─────────┐
│  Node A''  │ │ Node B   │ ← R2 的分支
│key:[5,6,7,8]│ │key:[9,10,
│ val: [1]   │ │    11,12]│
│ ref: 1     │ │val: [2]  │
└────────────┘ │ref: 1    │
               └──────────┘
```

---

### **步骤 3：请求完成与驱逐**
**R1 完成**：释放引用

#### unlock 后的引用计数：
```
         ROOT (ref=1, protected)
          │
    ┌─────▼──────┐
    │  Node A'   │
    │ ref: 1     │  ← 仅 R2 持有
    └──────┬──────┘
           │
      ┌────┴────┐
      │         │
┌─────▼──────┐ ┌▼─────────┐
│  Node A''  │ │ Node B   │
│ ref: 0     │ ← 可驱逐！  │ ref: 1     │
└────────────┘ └──────────┘
```

**R2 也完成**：
```
         ROOT (ref=1)
          │
    ┌─────▼──────┐
    │  Node A'   │
    │ ref: 0     │  ← 可驱逐
    └──────┬──────┘
           │
      ┌────┴────┐
      │         │
┌─────▼──────┐ ┌▼─────────┐
│  Node A''  │ │ Node B   │
│ ref: 0     │ │ ref: 0   │ ← 两个叶子都可驱逐
└────────────┘ └──────────┘
```

#### 驱逐策略（LRU）：
- 收集所有 `ref_count=0` 的叶子节点：Node A'', Node B
- 根据 `timestamp` 使用最小堆排序
- 假设 Node A'' 更老，优先驱逐
- 释放物理页 [1]

## 6. Naive Cache 与 Radix Cache 的对比

为了更好地理解 Radix Cache 的优势，我们将它与最基础的 `Naive Cache` (`python/minisgl/kvcache/naive_cache.py`) 进行对比。

### 6.1 Naive Cache 代码分析
在 `naive_cache.py` 中，我们可以看到它的实现非常“原始”：
- `match_prefix`：永远返回 `MatchResult(NaiveCacheHandle())`（命中长度为 0）。
- `insert_prefix`：什么都不做，返回插入长度 0。
- `get_matched_indices`：永远返回空的 Tensor。
- `evict`：不支持驱逐（直接抛出 `NotImplementedError`，说明它可能只依赖底层的页表分配器在请求结束时直接释放所有内存）。

**结论**：Naive Cache 完全**没有跨请求的状态保留**。一旦一个请求完成，它的 KV Cache 就被丢弃。下一个请求即使有完全相同的前缀，也必须从头计算（全量 Prefill）。

### 6.2 详细对比案例：多轮对话场景

假设用户进行了一次多轮对话：
- **Turn 1 (请求 R1)**: `System: You are a helpful AI. User: Hello! AI: Hi there.` (共 100 tokens)
- **Turn 2 (请求 R2)**: `System: You are a helpful AI. User: Hello! AI: Hi there. User: How are you?` (共 120 tokens，前 100 个与 R1 相同)

#### 在 Naive Cache 下的表现：
1. **R1 到达**：计算 100 tokens 的 Attention (Prefill)，生成回复。R1 结束后，释放这 100 tokens 的 KV Cache。
2. **R2 到达**：即使前 100 个 token 完全一样，由于缓存已清空，必须**重新计算**前 100 个 token 的 Attention，再加上新增的 20 个 token。
3. **代价**：随着对话轮数增加，每次的 Prefill 计算量呈**线性增长**（100 -> 120 -> 140...）。TTFT（首字延迟）会越来越长，严重浪费算力。

#### 在 Radix Cache 下的表现：
1. **R1 到达**：计算 100 tokens 的 Attention。结束后，这 100 tokens 的 KV Cache 以 `[System: You...]` 作为 Key 被保存在 Radix Tree 中。
2. **R2 到达**：`match_prefix` 发现前 100 个 token 完全命中现有的树节点！
3. **优化结果**：
   - 直接复用前 100 tokens 的 KV Cache 物理内存。
   - **只需要计算新增的 20 个 token 的 Attention (Prefill)**。
4. **代价**：无论对话进行多少轮，每次请求的 Prefill 计算量**仅仅等于用户新输入的内容长度**。TTFT 保持极低且稳定。

---

## 7. 核心优势总结

| **特性**               | **Naive Cache**            | **Radix Cache**           |
|------------------------|----------------------------|---------------------------|
| **跨请求缓存重用**     | 无 (请求结束即释放)        | 有 (通过基数树持久化)     |
| **前缀匹配能力**       | `prefix_len = 0` (全未命中)| 动态匹配最长前缀          |
| **R2 的 Prefill 长度** | 120 tokens (全量计算)      | 20 tokens (仅计算新内容)  |
| **TTFT (首 token 延迟)**| 随上下文长度线性增加       | 极低且不受历史长度影响    |
| **多轮对话显存占用**   | 峰值高，存在大量冗余计算   | 共享前缀，消除冗余计算    |
| **实现复杂度**         | 极低，无状态维护           | 高，需维护树结构、引用计数和LRU|

---

## 8. 总结
Radix Cache 的精妙之处在于将传统的 KV Cache 管理从简单的线性存储升级到了动态维护的树状结构，使得"前缀重用"这一 LLM 推理中的关键特性得以在系统层高效实现。通过对比 Naive Cache，我们可以看到 Radix Cache 解决了大模型推理中（尤其是多轮对话、Agent 场景）最大的计算瓶颈——**重复的 Prompt 计算**。它通过：
- **自动节点分裂**：当新请求部分匹配时，动态调整树结构
- **细粒度引用计数**：精确控制哪些缓存可驱逐
- **LRU 驱逐**：在显存不足时，优先释放最久未用的叶子节点
